01 · Rung 3 — Tools / MCP

What is MCP?

The Model Context Protocol is an open standard for connecting AI models to tools and data. You already wrote @tool functions — MCP is the agreed-upon plug shape so any app can use any tool, without custom glue.

Anthropic's analogy

"USB-C for AI." One connector instead of a different cable for every device. Write the server once; every MCP-aware app can plug in.

Open-sourced by Anthropic in late 2024, now adopted broadly across the industry.

01 · Why it exists

The M × N integration problem.

  • Every app × every tool = a bespoke integration to build and maintain.
  • MCP collapses it to M + N: each app speaks the protocol, each tool exposes it once.
  • The Kestrel ticketing system becomes one server — Claude Desktop, your IDE, and your agent all reach it the same way.

01 · Architecture

Host → Client → Server.

01 · Transports — how they talk

Local or remote.

stdio — local

  • Server runs as a subprocess on your machine
  • Talk over standard in/out — no network
  • Great for filesystem, local scripts, dev tools

Streamable HTTP — remote

  • Server runs somewhere else, reached over HTTP
  • The current standard for hosted/remote servers
  • Supports streaming and multiple clients
Version note

The older HTTP+SSE transport is now legacy — Streamable HTTP replaced it as the recommended remote transport (November 2025).

01 · The building blocks

A server exposes three things.

Tools
Model-invoked actions. The model decides to call them — create a Kestrel ticket, look up an order.
Like the @tool you wrote — now standardized.
Resources
Read-only data / context. Files, records, docs the app can load — the FAQ, a customer's device history.
Application-controlled, not called like a function.
Prompts
User-invoked templates. Reusable, parameterized workflows a person picks — "draft an escalation reply."
Surfaced as slash-commands / menu items.

Roughly: Tools = the model acts · Resources = the app supplies context · Prompts = the user triggers a flow.

01 · Primitive 1 — Tools

Tools — the model takes action.

A tool is a function the model can choose to call. The server advertises a name, a description, and a typed input schema — the model reads those and decides when to invoke it. Illustrative Kestrel tool.

# tool the Kestrel server advertises
{
  "name": "lookup_warranty",
  "description": "Get warranty status for a Kestrel device by SKU.",
  "inputSchema": {
    "type": "object",
    "properties": { "sku": { "type": "string" } },
    "required": ["sku"]
  }
}

Ask "is my Aria still under warranty?" and the model calls lookup_warranty("KH-AR-GLD") on its own — the 26-month clock lives in the server, not the prompt.

01 · Primitive 2 — Resources

Resources — read-only context.

A resource is data the server exposes for the app to read, addressed by a URI. Unlike tools, the model doesn't invoke them — the host or user decides what to load into context.

What it looks like

  • URI: kestrel://faq
  • Returns the support FAQ text as context
  • Others: kestrel://device/{id}/history

How it differs from a tool

  • App / user-controlled, not model-invoked
  • No side effects — a pure read
  • Think "attach a file", not "call a function"

Resources are how a server offers context; tools are how it offers actions.

01 · Primitive 3 — Prompts

Prompts — user-invoked templates.

A prompt is a reusable, parameterized message a person triggers — think slash-commands. The server ships the template; the user fills the blanks.

# prompt exposed by the Kestrel server
{
  "name": "support_reply",
  "description": "Draft a Kestrel-voice reply to a ticket.",
  "arguments": [{ "name": "ticket", "required": true }]
}

user types:  /support_reply ticket=#8842

Tools the model picks · resources the app supplies · prompts the user fires — three ways context reaches the model.

01 · Under the protocol

One tool call, on the wire.

MCP rides on JSON-RPC. When the model calls lookup_warranty, this is what actually crosses the connection. Illustrative.

// request — host → server
{ "method": "tools/call",
  "params": { "name": "lookup_warranty",
             "arguments": { "sku": "KH-AR-GLD" } } }

// result — server → host
{ "content": [{ "type": "text",
    "text": "Aria · covered · 26-month warranty · 14 months left" }] }

The model never touches your database — only this typed request and text result. That envelope is the whole standard.

01 · Write one yourself

Build a server in ~30 lines.

The FastMCP helper turns a decorated Python function into a full MCP server. Illustrative.

from mcp.server.fastmcp import FastMCP

mcp = FastMCP("kestrel")

@mcp.tool()
def lookup_warranty(sku: str) -> str:
    """Get warranty status for a Kestrel device by SKU."""
    return warranty_db.status(sku)

if __name__ == "__main__":
    mcp.run()

The docstring becomes the description; the type hints become the schema. You just made a tool any MCP agent can use.

01 · Remote servers & trust

Going remote — and staying safe.

stdio — local

  • Subprocess on your machine, no network
  • Runs with your permissions

Streamable HTTP — remote

  • Hosted elsewhere, reached over HTTP
  • OAuth 2.1 + MCP Gateways (2026) for auth
Security note

A server runs with your access, and a tool result can carry prompt injection back into the model. Vet untrusted servers before you connect — treat them like code you're about to run.

01 · Under the hood — the wire format

It's all JSON-RPC 2.0.

Every MCP message is one of three shapes. A request carries an id; a response echoes that same id; a notification has no id and expects no reply. Illustrative.

// request — has an id, expects a matching reply
{ "jsonrpc":"2.0", "id":7, "method":"tools/call",
  "params":{ "name":"lookup_warranty", "arguments":{...} } }

// response — same id 7, the answer
{ "jsonrpc":"2.0", "id":7, "result":{ "content":[...] } }

// notification — no id, fire-and-forget
{ "jsonrpc":"2.0", "method":"notifications/initialized" }

Methods you'll see: initialize · tools/list · tools/call · resources/list · resources/read · prompts/list · prompts/get.

01 · The lifecycle — how a session opens

The handshake, step by step.

Exactly the order the MCP Inspector shows: initialize → initialized → tools/list → tools/call. Nothing runs until both sides have shaken hands.

01 · The handshake's payload

Capability negotiation.

In that first exchange, each side declares what it supports and both agree on a protocol version. Neither assumes a feature the other lacks.

Server declares

  • tools — callable actions
  • resources — readable context
  • prompts — user templates

Client declares

  • sampling — server may borrow the model
  • roots — filesystem scopes it exposes
  • elicitation — server may prompt the user
Why it matters

A server that offers sampling only uses it if the client said it can — capabilities are a contract, negotiated once at initialize.

01 · The hidden cost

The tool-definition tax.

Every connected server injects all its tool definitions — name, description, full JSON schema — into the model's context on every request, before you say a word.

100–200k

tokens burned by a typical 5–10 server enterprise stack — spent before the first user word.

GitHub's MCP server alone is ≈ 17,600–55,000 tokens of tool definitions.

Why it hurts

That budget is gone from every turn — slower, costlier, and less room for the actual task. The fix is to stop loading what you're not using.

01 · The fix — progressive tool loading

Load a pointer, fetch the schema on demand.

  • Startup loads only a lightweight search interface, not every schema.
  • The agent fetches an individual tool's schema on demand — the moment it needs it.
  • Anthropic's Tool Search — GA Feb 2026.
The principle

Load a lightweight pointer; fetch the heavy body only when needed.

01 · You're watching it right now

This very session works this way.

The assistant building this deck has hundreds of tools available — but they're marked "deferred". Only their names are loaded. To use one, it first calls a ToolSearch tool to fetch that tool's full schema.

# at startup — names only, schemas NOT loaded
deferred: WebFetch, NotebookEdit, navigate, get_page_text, …

need to fetch a page →
ToolSearch("select:WebFetch")     # load THIS schema, now
→ WebFetch schema loaded · tool now callable

That is Tool Search / progressive loading, happening live — a lightweight index, with the heavy schema pulled in only at the moment of use.

01 · A second fix — code execution

Let the agent write code instead.

Present the MCP servers as a code API. The agent writes a short script that imports only the tools it needs — so only those definitions ever load — and crunches data in the sandbox before returning a small result. Illustrative.

# the agent writes this — imports ONE tool, not all of them
from servers.kestrel import lookup_warranty

skus = ["KH-AR-GLD", "KH-AR-BLK", "KH-PRO-2"]
covered = [s for s in skus if lookup_warranty(s).covered]
print(f"{len(covered)} of {len(skus)} still covered")   # small result

Only lookup_warranty's schema loads, and the loop's intermediate data never touches the model's context — just the one-line answer does.

01 · The third fix — trim at the server

Compress the schema, filter the fields.

Schema compression

  • Strip long descriptions and enum lists from tool defs
  • Keep just what the model needs to call it
  • Shrinks the always-loaded tax directly

Field filtering

  • Return only the 3–5 fields a workflow actually uses
  • Not the entire API response object
  • 80–90% payload cut on results
Same principle, server side

The client trims what it loads; the server trims what it sends. Both fight the same context bloat from opposite ends.

02 · Rung 4 — Agent Skills

What is an Agent Skill?

A folder the model can open when it's relevant. At minimum one file — SKILL.md — plus optional scripts and reference files bundled alongside.

Think of it as an onboarding packet for a specific task — how Kestrel handles escalations — that the model reads only when needed.

02 · The core mechanism

Progressive disclosure.

  • Tier 1 — just the name + one-line description sits in context, always.
  • Tier 2 — when the task matches, the model reads the full SKILL.md.
  • Tier 3 — it opens extra files or runs a script only if the job calls for it.
Why it matters

Context stays tiny until a skill is actually in play.

02 · The efficiency win

Why progressive disclosure pays off.

~140×

The always-loaded index for a skill is about 500 tokens. Loading all its docs up front could be ~70,000 tokens.

So the model can "know about" dozens of skills for a rounding error of context — and only pay the full cost for the one it actually uses.

The trade

A tiny table of contents, always present · the full chapter, only when opened.

02 · How a skill fires

The model picks it — like a tool.

Nobody flips a switch. The model reads each skill's description, matches it against the task, and decides to open the relevant one — the same selection instinct it uses to choose a tool.

  • So the description is load-bearing — write it like the trigger it is: "Use when handling a Kestrel support escalation."
  • A vague description means the skill never fires; a sharp one means it fires at exactly the right moment.
Same muscle you already trust

You already rely on the model to pick the right tool from a description. A skill is picked the very same way.

02 · Don't confuse them

MCP vs Skills.

MCP — a live connection

  • A link to an external system: a DB, an API, the ticketing app
  • Runs code in another process, over a protocol
  • Answers "what can the agent reach?"

Skills — packaged know-how

  • A folder of instructions (+ optional code) the model reads
  • Loads into context; can itself call MCP tools
  • Answers "how should the agent do this task?"
They're complementary

A Kestrel escalation skill can say "then call the create_ticket MCP tool at Tier 2." Knowledge guides; the connection acts.

02 · Anatomy

Inside a SKILL.md.

Frontmatter (always loaded) + Markdown body (loaded on match). Illustrative Kestrel skill.

---
name: kestrel-escalation
description: Route Kestrel support tickets. Use when a customer issue
  is unresolved >72h or mentions fire/smoke/burning/sparking.
---

# Escalation playbook

1. Safety words (fire, smoke, burning smell, sparking) → Tier 3 now,
   bypass the queue. Do not route manually.
2. Unresolved > 72h → auto-escalate to a Tier 2 specialist.
3. Otherwise open a Tier 2 ticket — owner: Marcus Webb, ext 4471.

See references/sla_table.md for full response-time targets.

02 · Anatomy of a skill

Frontmatter, body, and links.

A skill is a folder. SKILL.md carries metadata + instructions and can point to other files. Illustrative.

kestrel-support/
├─ SKILL.md
└─ reference.md

--- SKILL.md ---
---
name: kestrel-support
description: Answer Kestrel Home support questions on warranty,
  returns, and promo codes. Use for any Kestrel customer reply.
---
Follow the Kestrel voice. For exact policy numbers,
read reference.md (12% restocking, 26-month warranty).

Two fields are required — name and description. Everything else is prose the model reads only on a match.

02 · The one line that matters

The description is the trigger.

Claude never reads a skill's body until the description matches the task. That single line is the whole firing mechanism — so write it for when to use the skill, not what it contains.

Vague — never fires

  • "Kestrel support helper."
  • No trigger, no situation — the model can't tell when it applies

Sharp — fires on cue

  • "Use when a customer asks about Kestrel warranty, returns, or promo codes."
  • Names the exact moment to open it
Rule of thumb

Start the description with "Use when…" and name concrete triggers. That phrasing is what Claude matches on.

02 · Not just prose

Skills can run code.

A skill can bundle executable scripts next to its instructions. Claude runs them via bash when the task needs deterministic logic — code where code is better, prose where prose is better.

kestrel-support/
├─ SKILL.md
└─ scripts/
   └─ classify.py     # tier from ticket text

# SKILL.md says:
Run scripts/classify.py on the ticket to get the tier,
then reply.  (fire/smoke → Tier 3 · >72h → Tier 2)

The escalation rules become code that runs the same way every time — not something the model re-derives on each ticket.

02 · Executable skills, underneath

Where a bundled script runs.

A skill can ship scripts the model runs, not just reads. They execute in the code-execution environment of whatever surface is hosting the skill — the same sandbox the agent uses for any code.

Read (prose)

  • The model interprets the SKILL.md text
  • Flexible, but re-derived each run

  • Good for rules, math, parsing
The point

The tier logic becomes computed, not re-derived — a script that runs the same way every time, versioned right next to the prose.

03 · The order is deliberate

Why fine-tuning is last.

What fine-tuning costs you

  • You must curate a dataset — the real work
  • A training run + evaluation loop
  • Maintenance forever: retrain on every base-model upgrade
  • A new artifact to host, version, and monitor

What the rungs below cost

  • Edit a prompt — instant
  • Point RAG at a new doc — minutes
  • Add an MCP server — no model change
  • Drop in a skill folder — no model change

Everything below rung 5 changes around the model. Only fine-tuning changes the model — and that's the expensive part to keep alive.

03 · The mindset

Most problems never reach rung 5.

Land this sentence

Most problems are solved before you fine-tune — by a better prompt, the right context, a tool, or a skill.

Fine-tuning is a real and powerful tool. It is just the last one you reach for — when you've proven the cheaper rungs can't get you there.

04 · Rung 5 — Fine-tuning

You keep training the model.

Prompting, RAG, tools, skills never change the model — they change what surrounds it. Fine-tuning is different: you continue training on your own examples, and the model's weights actually move.

Everything below rung 5

  • Model weights stay frozen
  • You change the context around it
  • Reversible instantly — edit and re-run

Fine-tuning

  • Weights are updated by training
  • New default behavior baked into the model
  • Produces a new model artifact you own

04 · Three ways to do it

Full FT → LoRA → QLoRA.

Full fine-tune
Update every weight. Most powerful, most expensive — needs serious GPU hardware and storage.
LoRA
Freeze the base; train tiny adapter matrices instead. ~1% of the params, a small file to ship.
QLoRA
LoRA on a 4-bit quantized base. Fits training onto one consumer / free-T4 GPU.
The recommendation

Start with QLoRA. It's the cheapest thing that actually works — you can run it on a free Colab T4 today.

04 · How LoRA works

Train a small adapter, freeze the rest.

  • The big base matrix stays locked — you never touch it.
  • Two small matrices (rank r) learn the adjustment and add on top.
  • You ship just those adapters — megabytes, not gigabytes.
QLoRA's extra trick

Load that frozen base in 4-bit — same adapters, a fraction of the memory.

04 · The real work

The data is 80% of the job.

Training is a script you run. The hard, valuable part is curating examples — usually instruction / response pairs in JSONL. Quality beats quantity, badly.

// kestrel_support.jsonl — one example per line
{"instruction": "Customer wants to return an opened Aria, day 60.",
 "response": "Thanks for reaching out to Kestrel Home! Returns after
   the 45-day window carry a 12% restocking fee, waived if it's
   a confirmed hardware defect. Want me to check which applies?"}

Every example teaches the shape of a good answer — the greeting, the policy, the follow-up. That shape is what the model learns.

04 · The pipeline

Base → data → train → evaluate → deploy.

Hold out a slice of examples the model never sees in training. If it does well on those, it learned the pattern — not the answers.

04 · Real defaults

Sane starting hyperparameters.

04 · Inside one step

What one training step does.

Predict, measure the error, trace it back to the weights, nudge them down the gradient. That loop, thousands of times, is training.

04 · LoRA, a little deeper

Rank, alpha, and where it attaches.

  • rank r — how wide the adapter is (8–32). Bigger = more capacity and more params.
  • alpha — a scaling knob on how strongly ΔW is applied.
  • target modules — which layers get adapters (usually the attention projections).
Why it's portable

A and B are ~1% of the weights — a few MB you attach to, or detach from, the base at will.

04 · Did it actually work?

Evaluation, honestly.

How to check

  • Run the held-out set — data it never trained on
  • Format adherence: does every reply hit the 5-part template?
  • LLM-as-judge: a bigger model scores tone + correctness

Overfitting signals

  • Train loss → 0 but validation loss rises
  • Great on trained tickets, brittle on new ones
  • Repeats training answers verbatim
The one rule

A score on data the model trained on is marketing. Only the held-out set tells you the truth.

04 · Beyond SFT

DPO and GRPO, one line each.

DPO — preference

  • Trains on chosen vs rejected pairs
  • Teaches the model to prefer the better reply
  • Great for tone and judgment calls

GRPO — reinforcement

  • Rewards good outcomes, not fixed answers
  • Used to train reasoning behavior
  • Heavier; reach for it last
The mental model

SFT teaches the format · DPO teaches the preference · GRPO teaches the reasoning. Most support work stops at SFT.

04 · What "learning" actually is

Downhill, one step at a time.

  • The loss measures how wrong a prediction is.
  • The gradient points downhill; each step nudges the weights that way.
  • The learning rate is how big each step is — too big overshoots, too small crawls.
That's all training is

Predict, measure error, step downhill — repeat until the loss stops falling.

04 · The words on the dashboard

Epoch, batch, step, learning rate.

TermWhat it meansTypical
BatchHow many examples the model sees before one weight update2–8
StepOne forward + backward + update — one batch processed60–1000s
EpochOne full pass over the entire dataset1–3
Learning rateStep size down the gradient2e-4
The knob that bites

Too many epochs on a small set is the fastest way to overfit — the model runs out of new things to learn and starts memorizing.

04 · The math behind "1%"

Why LoRA trains so few parameters.

Freeze a weight matrix W (d×d). Instead of retraining it, learn a low-rank update ΔW ≈ B·A — where B is d×r and A is r×d, with rank r ≪ d.

# one 4096 × 4096 attention projection
full W        = 4096 × 4096       = 16,777,216 params
LoRA (r=16)   = 4096×16 + 16×4096 =    131,072 params

# ~0.8% of that one matrix — and you ship only B and A

Across the model that lands at roughly 0.1–1% of all parameters trained. alpha scales how hard ΔW is applied; target modules pick which layers (q / k / v / o / gate / up / down proj) get an adapter.

04 · How a big model fits a small GPU

QLoRA — quantize the frozen base.

  • Load the frozen base in 4-bit (NF4) — a quarter of the usual memory.
  • Train the LoRA adapters in higher precision on top of it.
  • A model that needed a data-center GPU now fits a free T4.
The unlock

Quantization shrinks the part you're not training; precision stays where the learning happens.

04 · Reading the loss curve

Where training goes wrong, on the chart.

  • Healthy: train loss settles around 0.5–1.0.
  • Overfitting: train loss dives toward 0 while validation rises.
  • Watch the train-vs-validation gap — not just the training line.
The tell

A loss near zero isn't success — it's the model memorizing the answers.

04 · Which model, which format

Instruct model + chat template.

You usually fine-tune an instruct model, not a raw base — it already knows how to follow turns. The chat template formats system / user / assistant into the exact token sequence it trains on. Illustrative.

# your data: a messages list
[{"role":"user", "content":"Return an opened Aria, day 60?"},
 {"role":"assistant", "content":"Thanks for reaching out…"}]

# the chat template renders it to tokens
<|user|>Return an opened Aria, day 60?<|end|>
<|assistant|>Thanks for reaching out…<|end|>

Use the same template the base model was trained with — a mismatched one is a common, silent way to wreck a run.

04 · What a preference example looks like

One DPO example: chosen vs rejected.

SFT shows the model a good answer. DPO shows it two — and which one to prefer. Illustrative.

{ "prompt": "Customer wants a refund past the return window.",

  "chosen": "Thanks for reaching out to Kestrel Home! Past the
     45-day window a 12% restocking fee applies — waived for a
     confirmed defect. Want me to check which applies?",

  "rejected": "No. Returns are only allowed within 45 days." }

Both are on-topic — but one is on-voice and helpful, the other blunt. DPO trains the model to lean toward the chosen shape.

04 · The risk of over-specializing

Catastrophic forgetting.

Push a model hard on one narrow task and it can erase general ability it used to have — the new weights overwrite the old skill. Full fine-tuning is most exposed; that's a big reason we reach for LoRA first.

How it happens

  • Too many steps on too narrow a dataset
  • Every weight shifts toward one kind of answer
  • Grammar, reasoning, other domains degrade

How to avoid it

  • LoRA — the base stays frozen, only adapters move
  • Limited steps — stop when the task is learned
  • Mix in some general examples alongside yours
The balance

You want the new voice and the old competence. Small updates on clean data keep both.
